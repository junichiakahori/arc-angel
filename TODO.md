## 📜 開発基本ルール (General Policy)
- [ ] **CHANGELOGの自動更新**: バグ修正や機能追加を行った際は、必ずユーザー向けに `CHANGELOG.md` にその内容を追記する。（明示的な指示がなくても自律的に行う）
- [ ] **バージョン管理フロー**: 
    - **開発中の編集禁止**: いかなるバグ対応・機能追加であっても、作業中は **絶対に `index.html` を直接編集しない** こと。（★重要：修正中は必ず `latest/` フォルダ内の最新バージョン `.html` のみを編集の対象とする）
    - すべての作業は `latest/vX.X.html` (現在は v0.9) のみで行い、動作確認完了後のリリース時のみ `index.html` へ反映する。
- [ ] **日時の厳密な更新（★必須・毎回言われずに行うこと）**: 開発版のファイルに少しでも変更を加えた場合、ユーザーから言われなくても、ファイル修正のタイミングで**必ず自律的に全履歴の日時を更新する**こと。
    - `assets/version_history.json`: 対象バージョン（`v0.9`等）の日時を**ユーザーの現在時刻（メタデータ参照）に合わせて**更新する。**（※絶対に未来時刻を記述しないこと。１分でも未来は不可）**
    - `latest/vX.X.html` (例: v0.9.html): 内部定数 `MY_VERSION` が JSON のキーと一致していることを確認する。
    - （※タイトルタグ `<title>` は JavaScript により JSON から動的に生成されるため、直接編集は不要になりました）
    - （変更を行った同一ターン内に、漏れなくこれらは完遂させること）
- [ ] **ファイル命名規則の統一化**: 
    - **ルート・ドキュメント**: `README.md`, `TODO.md`, `CHANGELOG.md` などは標準的な**全大文字 (UPPER CASE)** を維持。
    - **リソース・アセット**: 楽曲データ、画像、動画、JSON、スクリプト等はすべて **スネークケース (`snake_case`)** に統一し、大文字・キャメラケースを排除する。
- [ ] **楽曲の並び順**: リスト (`songs.json`) への楽曲追加は、必ず μ's のリリース日順（歴史順）に行うこと。
- [ ] **譜面アレンジの高度化**: HARD以上の難易度では、単押し・同時押しに加えて **「ロングノーツ（長押し）」** を積極的に取り入れ、リズムの多様性を出すこと。
- [ ] **進捗の同期**: 作業の節目で必ず Git のコミットと GitHub へのプッシュを確認する。
- [ ] **修正後の自動プッシュ**: バグ修正や機能追加、ソースコードの変更を行ったあとは、ユーザーから言われなくても**必ず自身のターミナルツールを用いて `git add . && git commit -m "..." && git push` を実行する** こと。（不要な確認待ちを減らし、シームレスな体験を提供する）
